# Flake-native sbtix example

This directory is a minimal Scala application that shows the recommended
**flake-native** workflow:

1. Add sbtix as a flake input.
2. Regenerate `repo.nix` lockfiles with `sbtix-gen`.
3. Build the app from your flake with `buildSbtProgram`.

It does **not** use `sbtix-generated.nix`. The flake imports sbtix's Nix helpers
directly from the flake input.

## Quick start

```bash
cd examples/flake-app
nix build
./result/bin/flake-app-example
```

Expected output:

```text
Hello from sbtix flake-app example
```

## Regenerating dependency locks

When dependencies change, refresh the lockfiles from the dev shell:

```bash
nix develop
sbtix-gen
git diff repo.nix project/repo.nix project/project/repo.nix
```

Then rebuild:

```bash
nix build
```

## Layout

| File | Purpose |
|------|---------|
| `flake.nix` | Dev shell + `packages.default` using `buildSbtProgram` |
| `repo.nix` | Main dependency lockfile (generated by `sbtix-gen`) |
| `project/repo.nix` | Meta-build lockfile |
| `project/project/repo.nix` | Meta-meta-build lockfile |
| `manual-repo.nix` | Optional manual artifact pins |

## Developing against a local sbtix checkout

When working inside the sbtix repository, point the flake input at the parent
checkout:

```bash
nix build --override-input sbtix "path:../.."
```

Or change `inputs.sbtix.url` in `flake.nix` to `path:../..` while developing.

## Copying into your own project

1. Copy the Scala sources, `project/`, lockfiles, and `flake.nix`.
2. Keep the pinned `sbtix` input (or update the tag).
3. Adjust `packages.default` if you need `buildSbtLibrary` or a custom
   `buildSbtProject` install phase.

See also [`../../flake.nix.example`](../../flake.nix.example) for a template
you can drop into an existing flake.
